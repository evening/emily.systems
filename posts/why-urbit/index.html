why urbit?

September 25, 2022

I can’t stop thinking about Urbit. I wanted to write about how it could change the way we build and use the internet.

Design example

Let’s build Snapchat â€“ we want to send photos to friends. How would it typically be built, and how could we built it in Urbit?

Handling users

The service needs authentication. This would require storing the usernames and passwords in a database. We’d also have to design some sort of password recovery system.

With Urbit, the OS handles authentication. If you are logged in to the Urbit OS, you are already authenticated. It’s like SSO, except it’s your own.

Sending data

Then, we have to handle the actual sending. Each time someone adds a friend, we’d have to store it to a database. Each time someone posts a photo, the photo gets uploaded to our storage and gets written to a database so each friend can view it.

With Urbit, your own machine stores your friends list. Each time you post a photo, it reads your own friends list and sends it directly to them (Urbit handles the peer-to-peer network on the OS level: simply send it to the Urbit ID). The photo only gets delivered to those you want it to.

Scaling

Let’s say your service becomes a lot more active than anticipated; we’d have to scale up the service. This could include increasing the amount of storage for photos, increasing the amount of uploads the server can handle at once, replication, sharding, and so on. This would increase complexity and the cost required to run the service. We would probably need to monetize our service. Maybe ads?

With Urbit, there is no need to scale. Everything is peer to peer. Data isn’t being sent to a single centralized server but to your friends.

Shut down

Let’s say you don’t want to run the service anymore: it’s been fun, but it’s too expensive. You shut it off, and all consumers are affected.

With Urbit, there is nothing to ‘shut down’. It’ll exist as long as people use it.
